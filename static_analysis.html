Hồ sơ 00 · Overview

Static Analysis là gì, và vì sao pentester luôn bắt đầu từ đây?

Trước khi cắm cáp, cài Frida hay bật Burp Suite lên soi traffic, hầu hết dân audit mobile app đều làm một việc rất "nhàn": mở gói cài đặt ra đọc — mà không chạy app một lần nào. Đó chính là static analysis.

Static vs Dynamic Analysis

Bấm để so sánh hai cách tiếp cận
Định nghĩa
Kiểm tra file .apk / .ipa mà không chạy ứng dụng — đọc code đã decompile, resource, file cấu hình, permission, chuỗi ký tự...
Công cụ
jadx, apktool, MobSF, Ghidra — không cần app đang chạy.
Ưu điểm
An toàn (không đụng hệ thống thật), không cần rooted device/emulator, phát hiện sớm hardcoded secret, config sai, permission thừa.
Hạn chế
Không thấy được hành vi runtime thực tế; code bị obfuscate nặng sẽ khó đọc hơn nhiều.
Định nghĩa
Chạy app trên thiết bị thật hoặc emulator và quan sát hành vi thực tế: network traffic, file I/O, API call, giá trị biến trong lúc chạy.
Công cụ
Frida, Objection, Burp Suite / mitmproxy — cần app đang chạy.
Ưu điểm
Thấy hành vi thật, có thể hook hàm để bypass một số cơ chế bảo vệ runtime (ví dụ SSL pinning).
Hạn chế
Cần môi trường (device/emulator, đôi khi phải root), tốn thời gian hơn, có thể bị anti-debugging / anti-root chặn lại.

💡 Trong một bài test mobile app thực tế, hai kỹ thuật này không loại trừ nhau — static analysis thường được làm trước để định hướng, dynamic analysis làm sau để xác thực lại giả thuyết.

Ví dụ minh hoạ

Trong một buổi đánh giá bảo mật giả định cho ứng dụng ngân hàng NovaBank, chỉ bằng static analysis — chưa bật app lên lần nào — pentester đã phát hiện: access token được lưu plaintext trong SharedPreferences, cờ android:debuggable="true" vẫn còn bật ở bản release, và một API key bị hardcode ngay trong AndroidManifest.xml. NovaBank là ví dụ giả định xuyên suốt tài liệu này — mọi mã nguồn, key, giá trị đều là dữ liệu minh hoạ, không phải của ứng dụng thật.

👆 Bấm vào gói tin để "bóc niêm phong"

Một file .apk / .ipa chỉ là một túi bằng chứng đã niêm phong

Bên trong nó là ZIP archive chứa code đã biên dịch, resource, file cấu hình và chữ ký số. Toàn bộ hành trình static analysis, về cơ bản, chỉ là quá trình mở túi bằng chứng này ra và đọc từng thứ bên trong một cách có hệ thống.

  • META-INF/ — chữ ký số của app
  • AndroidManifest.xml — cấu hình lõi
  • classes.dex — bytecode để decompile

Tài liệu này tập trung vào static analysis. Các kỹ thuật dynamic analysis (Frida, Objection, SSL pinning bypass...) chỉ được nhắc đến để đối chiếu, không đi sâu.

Hồ sơ 01 · APK / IPA

Bên trong gói tin có gì?

Một file .apk (Android) hay .ipa (iOS) đều thực chất là một file ZIP — đổi đuôi thành .zip là giải nén được ngay bằng bất kỳ trình giải nén nào. Bấm vào từng file/thư mục bên dưới để xem vì sao pentester quan tâm đến nó.

👈 Bấm vào 1 file hoặc thư mục bên trái để xem chi tiết tại đây.

NovaBank.apk

Đây là file gốc — thực chất là 1 ZIP archive. Có thể giải nén trực tiếp bằng unzip, hoặc dùng apktool để decode luôn cả resource và Manifest sang dạng đọc được.

Việc đầu tiên khi nhận 1 APK: kiểm tra hash (MD5/SHA-256) để đối chiếu đúng bản build đang test, tránh nhầm phiên bản.

META-INF/

Thư mục chứa toàn bộ thông tin ký số (code signing) của gói APK.

Vì sao quan tâm: cho biết ai đã build & ký app này, và app là bản debug-signed hay release-signed — một app debug-signed lọt ra ngoài production là dấu hiệu quy trình release có vấn đề.

META-INF/MANIFEST.MF

Liệt kê tên file và giá trị hash (digest) của từng file trong package — nền tảng để xác minh tính toàn vẹn.

Vì sao quan tâm: nếu 1 file trong APK bị chỉnh sửa sau khi ký mà digest không khớp, hệ thống cài đặt sẽ từ chối — đây là cơ chế chống tampering cơ bản của Android.

META-INF/CERT.SF

Chữ ký số trên chính nội dung của MANIFEST.MF — bước xác thực nối tiếp trong chuỗi ký.

Vì sao quan tâm: cùng với CERT.RSA, đây là bằng chứng "app này đúng là do bên giữ private key ký" — hữu ích khi cần xác minh nguồn gốc 1 bản APK nghi ngờ.

META-INF/CERT.RSA

Certificate + public key dùng để xác minh chữ ký ở trên.

Vì sao quan tâm: trích xuất được thông tin issuer, ngày hết hạn certificate — nếu certificate dùng để ký là self-signed với thời hạn rất dài hoặc thông tin issuer bất thường, đó có thể là dấu hiệu app giả mạo (fake app) khi so với bản chính hãng.

res/

Resource ở dạng đã compile một phần: layout, drawable, một số giá trị string.

Vì sao quan tâm: layout đôi khi tiết lộ tên field/API ẩn (ví dụ 1 màn hình debug chưa gỡ), và res/xml/network_security_config.xml — file cấu hình bảo mật mạng — nằm ở đây.

assets/

Nơi developer nhét file thô tuỳ ý — không bị Android build tool xử lý/nén như res/.

Vì sao quan tâm: đây là "ngăn kéo hay bị quên" — pentester thường tìm thấy file config JSON, database mẫu, thậm chí cả private key hoặc file .env bị lỡ đóng gói vào đây.

lib/

Chứa native library dạng .so, biên dịch từ C/C++ (qua NDK), tổ chức theo kiến trúc CPU.

Vì sao quan tâm: logic nhạy cảm (crypto, chống dịch ngược, chống root) hay được đẩy xuống native code để gây khó cho reverse engineer — cần Ghidra hoặc IDA để phân tích ở mức assembly.

lib/arm64-v8a/, armeabi-v7a/, x86/

Bản build riêng của cùng 1 thư viện native cho từng kiến trúc CPU (thiết bị thật thường là arm64-v8a/armeabi-v7a, emulator hay dùng x86/x86_64).

Vì sao quan tâm: chọn đúng kiến trúc khớp với thiết bị/emulator đang test khi load thư viện vào Ghidra, tránh phân tích nhầm bản.

AndroidManifest.xml

File cấu hình lõi của app: package name, permission, danh sách component (Activity/Service/Receiver/Provider), cờ bảo mật... nhưng ở dạng binary XML, không đọc trực tiếp được.

Vì sao quan tâm: đây gần như là "bảng điều khiển bề mặt tấn công" của app — xem chi tiết ở Hồ sơ 03. Cần apktool hoặc aapt để decode ra dạng text đọc được.

classes.dex

Dalvik bytecode — chính là "code" của app sau khi Java/Kotlin source được biên dịch.

Vì sao quan tâm: đây là thứ ta decompile (bằng jadx hoặc dex2jar) để đọc lại gần như source code gốc — nơi tìm business logic, hardcoded secret, thuật toán mã hoá tự chế...

classes2.dex (multidex)

Khi app có hơn 65.536 method (rất phổ biến với app lớn như banking app), Android build tool phải tách bytecode thành nhiều file .dex.

Vì sao quan tâm: đừng quên decompile hết tất cả classesN.dex — business logic quan trọng có thể nằm ở file thứ 2, thứ 3 chứ không chỉ classes.dex đầu tiên.

resources.arsc

Bảng ánh xạ resource đã compile: resource ID → giá trị (string, màu, dimension...).

Vì sao quan tâm: chuỗi trong strings.xml gốc cuối cùng cũng nằm ở đây dưới dạng compiled — công cụ trích xuất string hàng loạt (ví dụ apkleaks) thường đọc thẳng từ file này.

NovaBank.ipa

Cũng là 1 ZIP archive, nhưng cấu trúc bên trong khác hẳn Android.

Vì sao quan tâm: giải nén bằng unzip là thấy ngay thư mục Payload/ — điểm bắt đầu của mọi phân tích IPA.

Payload/

Thư mục chuẩn chứa bundle ứng dụng thực sự bên trong.

Vì sao quan tâm: mọi thứ đáng phân tích đều nằm trong thư mục con .app/ bên trong đây.

NovaBank.app/

Bundle chính của ứng dụng — tương đương "ruột" của app.

Vì sao quan tâm: đây là nơi chứa binary thực thi, file cấu hình, và toàn bộ resource — tương tự vai trò tổng hợp của APK bên Android.

Info.plist

File cấu hình lõi — vai trò tương đương AndroidManifest.xml bên Android: bundle ID, permission usage description (ví dụ NSCameraUsageDescription), URL scheme, cấu hình ATS (App Transport Security).

Vì sao quan tâm: nếu ATS bị nới lỏng (cho phép NSAllowsArbitraryLoads) thì app có thể gửi traffic không mã hoá — tương đương với usesCleartextTraffic bên Android.

NovaBank (Mach-O executable)

Binary chính, biên dịch từ Swift/Objective-C.

Vì sao quan tâm: dùng class-dump để lấy thông tin class/method Objective-C, hoặc Hopper/Ghidra/IDA để reverse engineering sâu hơn — Swift thuần khó đọc hơn Objective-C nhiều vì tên bị "mangle".

embedded.mobileprovision

Hồ sơ provisioning — cho biết app được build theo kiểu nào.

Vì sao quan tâm: phân biệt bản Development / Ad Hoc / App Store / Enterprise, và liệt kê entitlement app được cấp — entitlement quá rộng cũng là một finding đáng ghi nhận.

_CodeSignature/

Chữ ký số của app — vai trò tương đương thư mục META-INF/ bên Android.

Vì sao quan tâm: xác minh app có bị chỉnh sửa sau khi ký hay không (tương tự cơ chế chống tampering của Android).

Frameworks/

Chứa các .framework/.dylib mà app phụ thuộc vào (thư viện bên thứ ba, SDK phân tích, SDK thanh toán...).

Vì sao quan tâm: SDK bên thứ ba lỗi thời cũng là bề mặt tấn công — nên liệt kê và đối chiếu version với CVE đã biết.
Hồ sơ 02 · Workflow

Quy trình static analysis, từng bước một

Đây là thứ tự phổ biến khi audit 1 mobile app — không phải luật cứng, nhưng đủ để bắt đầu có hệ thống thay vì mò mẫm. Bấm từng bước để xem lệnh & output minh hoạ.

Lấy được file .apk/.ipa cần test — từ thiết bị qua adb, từ bản build client cung cấp, hoặc qua kênh phân phối hợp lệ khi đã có ủy quyền test.

Giải nén gói tin ra để bắt đầu soi cấu trúc — xem lại Hồ sơ 01 nếu chưa rõ từng file bên trong là gì.

Chuyển bytecode/binary trở lại gần với source code đọc được. Với Android có 2 lựa chọn phổ biến: jadx (đọc nhanh, GUI trực quan) hoặc apktool (khi cần cả resource gốc + smali để rebuild lại app).

Đọc kỹ AndroidManifest.xml (Android) hoặc Info.plist (iOS) — gần như một "bản đồ bề mặt tấn công" thu nhỏ của app. Xem chi tiết ở Hồ sơ 03.

Đọc code đã decompile, đặc biệt các class liên quan auth, lưu trữ, mạng — kết hợp tìm kiếm hàng loạt theo pattern để không bỏ sót hardcoded secret. Thực hành ở Hồ sơ 04.

Kiểm tra network_security_config.xml, cách app dùng Cipher/MessageDigest, và có custom TrustManager nào bỏ qua việc verify certificate hay không.

Không có terminal cho bước này — đây là lúc viết finding: mô tả vấn đề, mức độ nghiêm trọng (severity), bằng chứng (đoạn code/dòng Manifest cụ thể), và khuyến nghị khắc phục rõ ràng, có thể hành động được.

            Info
            
package / versionCode / versionName

            
Định danh gói và số phiên bản build. Không phải lỗ hổng, nhưng luôn ghi lại — versionCode giúp đối chiếu app có đang chạy 1 version cũ với lỗ hổng đã biết (CVE) hay không.

          

          

            Medium
            
minSdkVersion / targetSdkVersion thấp

            
minSdkVersion="19" nghĩa là vẫn hỗ trợ Android 4.4 — thiếu rất nhiều cơ chế bảo vệ nền tảng hiện đại. targetSdkVersion="24" thấp cũng khiến app không được áp các mặc định an toàn hơn mà Android bản mới yêu cầu (ví dụ network security config chỉ bắt buộc từ API 24 trở lên theo mặc định khắt khe hơn ở các bản sau).

          

          

            Medium
            
Over-permission — READ_SMS

            
Một app ngân hàng xin quyền đọc SMS (thường viện lý do "tự động điền OTP") nhưng không có giải trình rõ ràng trong Manifest là dấu hiệu over-permission. Permission càng rộng, thiệt hại càng lớn nếu app hoặc 1 thành phần của nó bị khai thác.

          

          

            High
            
android:allowBackup="true"

            
Cho phép adb backup lấy toàn bộ dữ liệu app ra — kể cả khi thiết bị chưa root. Với 1 app lưu token/session trong SharedPreferences hoặc database nội bộ, đây là đường tắt để trích xuất dữ liệu đó ra phân tích offline.

          

          

            Critical
            
android:debuggable="true"

            
Cho phép attach debugger vào tiến trình app ngay ở bản release. Kẻ tấn công có thể debug trực tiếp, đọc bộ nhớ runtime, đặt breakpoint để bypass logic kiểm tra (ví dụ màn hình xác thực). Đây gần như luôn là lỗi cấu hình build, không bao giờ nên xuất hiện ở bản production.

          

          

            High
            
android:usesCleartextTraffic="true"

            
Cho phép app gửi traffic qua HTTP thuần (không mã hoá) đến bất kỳ domain nào chưa bị chặn riêng. Nếu app lỡ gọi dù chỉ 1 endpoint qua HTTP, dữ liệu trên đường truyền có thể bị đọc/sửa qua tấn công Man-in-the-Middle.

          

          

            Info / Tín hiệu tốt
            
android:networkSecurityConfig

            
Đây là điểm đáng khen — app có khai báo file cấu hình bảo mật mạng riêng. Nhưng đừng dừng lại ở đây: phải mở tiếp res/xml/network_security_config.xml để kiểm tra, vì file này hoàn toàn có thể chứa cấu hình cho phép cleartext theo từng domain, hoặc trust-anchor không phù hợp.

          

          

            High
            
Exported Activity + Deep Link

            
Activity được export ra ngoài (exported="true") kèm intent-filter nhận deep link theo scheme novabank://. Nếu Activity này không validate kỹ dữ liệu đầu vào từ Intent, một app khác (kể cả app độc hại đã cài trên cùng máy) có thể gọi thẳng vào Activity này, có nguy cơ bypass luồng xác thực hoặc kích hoạt hành động ngoài ý muốn.

          

          

            Critical
            
ContentProvider export không kiểm soát

            
exported="true" cùng grantUriPermissions="true" mà không kèm permission hay path-permission nào — app khác trên cùng thiết bị có thể đọc/ghi dữ liệu qua Provider này. Đây là một trong những nguồn phổ biến nhất dẫn tới path traversal hoặc rò rỉ file nội bộ trên Android.

          

          

            Critical
            
Hardcoded API key trong Manifest

            
Key nằm thẳng trong file mà bất kỳ ai giải nén APK cũng đọc được. Tuỳ vào quyền hạn của key (ví dụ quota của 1 dịch vụ bản đồ), hậu quả có thể từ bị lạm dụng quota, phát sinh chi phí, cho tới nguy hiểm hơn nếu key đó có phạm vi quyền rộng hơn cần thiết.

Hồ sơ 05 · Common Vulnerabilities

      
8 nhóm lỗ hổng hay gặp nhất qua static analysis

      
Bấm vào từng mục để mở rộng: mô tả, ví dụ code (minh hoạ), dấu hiệu nhận biết khi đọc code tĩnh, và hướng khắc phục. Nhãn góc phải tham chiếu nhóm trong OWASP MASVS.


      

        Hardcoded Secrets & API Keys ›
        

          MASVS-STORAGE
          
Ví dụ (minh hoạ)

          
private static final String API_KEY = "sk_live_51H8xxxxxxxxxxxxxxxx";

          
Dấu hiệu qua static analysis

          
Search string bất thường / entropy cao trong code đã decompile hoặc trong resource; các tiền tố quen thuộc như AKIA (AWS), AIza (Google), sk_live_ (Stripe) rất dễ grep.

          
Khắc phục

          
Đưa secret về backend, không bao giờ hardcode secret nhạy cảm ở phía client; nếu bắt buộc phải có key ở client, dùng Android Keystore hoặc inject lúc build và giới hạn quyền hạn của key tối thiểu.

        

      


      

        Insecure Data Storage ›
        

          MASVS-STORAGE
          
Ví dụ (minh hoạ)

          
SharedPreferences sp = getSharedPreferences("auth", MODE_PRIVATE);
sp.edit().putString("access_token", token).apply();

          
Dấu hiệu qua static analysis

          
Grep getSharedPreferences, SQLiteDatabase; sau khi giải nén data thư mục app (trên thiết bị root hoặc qua backup), mở trực tiếp file .xml/.db xem có plaintext không.

          
Khắc phục

          
Mã hoá dữ liệu nhạy cảm bằng EncryptedSharedPreferences hoặc lưu qua Android Keystore, không lưu token/mật khẩu ở dạng đọc được trực tiếp.

        

      


      

        Insecure WebView Configuration ›
        

          MASVS-PLATFORM
          
Ví dụ (minh hoạ)

          
webView.getSettings().setJavaScriptEnabled(true);
webView.addJavascriptInterface(new JSBridge(), "Android");

          
Dấu hiệu qua static analysis

          
Grep addJavascriptInterface, setJavaScriptEnabled(true), và các lệnh loadUrl nhận input động (từ Intent, deep link...).

          
Khắc phục

          
Hạn chế tối đa JavascriptInterface, đánh dấu method public bằng @JavascriptInterface có chọn lọc, validate origin trước khi load nội dung không tin cậy.

        

      


      

        Weak / Broken Cryptography ›
        

          MASVS-CRYPTO
          
Ví dụ (minh hoạ)

          
Cipher c = Cipher.getInstance("AES/ECB/PKCS5Padding");

          
Dấu hiệu qua static analysis

          
Grep Cipher.getInstance tìm chế độ ECB (không có IV, để lộ pattern dữ liệu), hoặc MessageDigest.getInstance("MD5")/"SHA1" dùng cho việc băm mật khẩu.

          
Khắc phục

          
Dùng AES/GCM với IV ngẫu nhiên cho mỗi lần mã hoá; dùng bcrypt/Argon2 cho việc lưu trữ mật khẩu, không dùng hash thường cho mục đích này.

        

      


      

        Insecure Network Communication ›
        

          MASVS-NETWORK
          
Ví dụ (minh hoạ)

          
public void checkServerTrusted(X509Certificate[] chain, String authType) {
    // để trống — không throw exception nào cả
}

          
Dấu hiệu qua static analysis

          
Tìm usesCleartextTraffic, custom X509TrustManager/HostnameVerifier luôn trả về true hoặc có thân hàm rỗng bất thường.

          
Khắc phục

          
Enforce HTTPS cho toàn bộ traffic, cân nhắc certificate pinning cho các API nhạy cảm, không bao giờ override logic verify certificate để "cho tiện lúc dev" rồi quên gỡ ở bản release.

        

      


      

        Excessive / Dangerous Permissions ›
        

          MASVS-PLATFORM
          
Dấu hiệu qua static analysis

          
Liệt kê toàn bộ <uses-permission> trong Manifest, đối chiếu với chức năng thực tế của app — permission nào không giải thích được lý do nghiệp vụ là điểm cần hỏi lại đội phát triển.

          
Khắc phục

          
Áp dụng nguyên tắc least privilege: chỉ xin permission thật sự cần thiết, và xin theo kiểu runtime request đúng lúc cần dùng thay vì khai báo tràn lan.

        

      


      

        Exported Components Without Protection ›
        

          MASVS-PLATFORM
          
Dấu hiệu qua static analysis

          
Soi từng Activity/Service/Receiver/Provider trong Manifest, tìm exported="true" mà không đi kèm android:permission hoặc kiểm tra caller trong code.

          
Khắc phục

          
Đặt exported="false" nếu component chỉ dùng nội bộ; nếu bắt buộc export, bảo vệ bằng custom permission ở mức signature.

        

      


      

        Thiếu Code Obfuscation / Tamper Protection ›
        

          MASVS-RESILIENCE
          
Dấu hiệu qua static analysis

          
Mở bằng jadx thấy code sạch đẹp, tên class/method rõ ràng kiểu LoginActivity.validateOtp() — gần như đọc được nguyên bản logic nghiệp vụ.

          
Khắc phục

          
Bật R8/ProGuard ở chế độ full obfuscation cho bản release; với app nhạy cảm (banking, ví điện tử) cân nhắc thêm lớp anti-tampering/root-detection, dù cần lưu ý các cơ chế này cũng chỉ làm chậm attacker chứ không ngăn tuyệt đối.

        

      

    


    
    

      
Hồ sơ 06 · Code Review Quiz

      
Thử thách: soi mã nguồn NovaBank

      
5 đoạn code minh hoạ, mỗi đoạn có 1 câu hỏi trắc nghiệm. Chọn đáp án để xem giải thích ngay lập tức.


      

        


        

          
Câu 1 / 5

          
<application
    android:allowBackup="true"
    android:debuggable="false"
    android:usesCleartextTraffic="false">

          
Thuộc tính nào trong đoạn Manifest trên cho phép trích xuất toàn bộ dữ liệu app qua adb backup mà không cần root?

          

            
            
            
            
          

          
Chính xác là android:allowBackup. Khi để "true", adb backup có thể lấy toàn bộ dữ liệu app ra file, kể cả trên thiết bị chưa root — rất hữu ích cho attacker nếu app lưu token/session không mã hoá.

          
        


        

          
Câu 2 / 5

          
Cipher c = Cipher.getInstance("AES/ECB/PKCS5Padding");
c.init(Cipher.ENCRYPT_MODE, secretKey);

          
Chế độ mã hoá nào trong đoạn code trên bị xem là yếu / không an toàn cho dữ liệu nhạy cảm?

          

            
            
            
            
          

          
Chính xác là AES/ECB. Chế độ ECB mã hoá từng block độc lập, không dùng IV, nên các block dữ liệu giống nhau sẽ cho ra ciphertext giống nhau — làm lộ pattern của dữ liệu gốc dù đã "mã hoá".

          
        


        

          
Câu 3 / 5

          
webView.getSettings().setJavaScriptEnabled(true);
webView.addJavascriptInterface(new JSBridge(), "Android");
webView.loadUrl(getIntent().getStringExtra("target_url"));

          
Đoạn code trên thuộc nhóm lỗ hổng nào?

          

            
            
            
            
          

          
Chính xác là Insecure WebView Configuration. Kết hợp addJavascriptInterface với việc load URL lấy trực tiếp từ Intent (không validate) tạo nguy cơ JavaScript injection và lộ luôn native interface cho nội dung không tin cậy.

          
        


        

          
Câu 4 / 5

          
<provider
    android:name=".FileProvider"
    android:exported="true"
    android:grantUriPermissions="true" />

          
Rủi ro chính của cấu hình ContentProvider trên là gì?

          

            
            
            
            
          

          
Chính xác là phương án đầu tiên. Provider export ra ngoài mà thiếu permission hoặc path-permission kiểm soát là một trong những nguồn phổ biến nhất dẫn tới rò rỉ dữ liệu hoặc path traversal trên Android.

          
        


        

          
Câu 5 / 5

          
public void checkServerTrusted(X509Certificate[] chain, String authType) {
    // để trống — không throw exception nào cả
}

          
Đoạn code này khiến ứng dụng dễ bị tấn công nào nhất?

          

            
            
            
            
          

          
Chính xác là Man-in-the-Middle. Một X509TrustManager với thân hàm rỗng chấp nhận mọi certificate, kể cả certificate giả mạo — kẻ tấn công ở giữa đường truyền có thể đọc/sửa dữ liệu mà app không hề hay biết.

Hồ sơ 07 · Toolbox

      
Bộ công cụ cho static analysis

      
Lọc theo nền tảng để xem công cụ nào phù hợp với việc bạn đang làm.


      

        
        
        
        
      


      

        
jadx / jadx-gui Android
Decompile .dex → Java source đọc được gần như code gốc. GUI trực quan, search toàn project nhanh. Hạn chế: code obfuscate nặng vẫn khó đọc.

        
apktool Android
Decode resource + Manifest ra dạng đọc được, kèm smali code. Dùng khi cần rebuild/patch lại APK, ví dụ thêm log debug hoặc vô hiệu hoá tạm 1 check để phục vụ test.

        
dex2jar + JD-GUI Android
Chuyển .dex → .jar rồi mở bằng JD-GUI để xem lại Java source. Cách làm truyền thống, trước khi jadx phổ biến.

        
MobSF Đa nền tảng
Tự động hoá cả static & dynamic analysis cho APK/IPA, sinh báo cáo nhanh gồm permission, quét hardcoded secret cơ bản, phát hiện lỗi Manifest phổ biến — tốt để có bức tranh tổng quan trước khi đào sâu thủ công.

        
Ghidra Đa nền tảng
Reverse engineering binary/native library (.so của Android, Mach-O của iOS) ở mức assembly — cần khi logic nhạy cảm bị đẩy xuống native code.

        
Androguard Android
Thư viện/CLI Python để phân tích APK theo hướng lập trình — script hoá việc trích Manifest, permission, hoặc tìm pattern hàng loạt qua nhiều file APK cùng lúc.

        
apkleaks Android
Quét tự động URL/API key/secret theo pattern phổ biến trong APK đã giải nén — tăng tốc đáng kể bước "săn bí mật" ở Hồ sơ 04.

        
TruffleHog Đa nền tảng
Quét secret theo entropy & regex pattern, dùng được trên source code đã decompile hoặc cả lịch sử git nếu có.

        
class-dump iOS
Trích xuất thông tin class/method Objective-C từ binary Mach-O — vai trò gần giống jadx bên Android, nhưng hạn chế nhiều với code Swift thuần.

        
plutil iOS
Đọc/convert file Info.plist (thường ở dạng binary plist) sang XML để review permission usage description, URL scheme, cấu hình ATS.
